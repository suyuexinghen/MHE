# Harness 设计原则：组装理论与抽象谬误

## 摘要

本文将组装理论与“抽象谬误”两套思想转化为 AI harness（框架/执行环境）的设计原则。组装理论说明：高复杂度产物若要可信，必须具备可追溯的制造历史、递归复用路径和足够高的拷贝数；抽象谬误说明：符号计算无论多复杂，都只是地图，不能自动等同于真实世界中的实例化行动或经验。二者合在一起，为 harness 提供了一套双重约束：一方面，harness 必须成为一个带物理记忆的选择环境，让复杂 agent、工具和策略通过历史积累逐步演化出来；另一方面，harness 必须成为一个诚实的制图系统，明确区分模拟、计划、声明与真实执行。

因此，一个成熟的 harness 不应只追求“让 agent 更聪明”，而应回答一个更基础的问题：如何稳定地产生、保存、验证和复用复杂能力，同时防止系统把无谱系复杂性误认为可靠能力，或把内部符号模拟误认为外部世界事实。

## 一、核心结论

Harness 的目标不是让单个 agent 自身具有高组装指数，而是让 harness 成为一个能够持续演化出高组装指数、高拷贝数产物的选择环境。

更准确地说，harness 应被设计成一个“带记忆的制图工坊”：

1. **带记忆**：保留被验证有效的工具、策略、流程、评估结果和失败记录。
2. **可递归**：允许复杂能力通过已有组件的递归复用逐步堆叠，而不是从零随机生成。
3. **有选择压**：持续淘汰低复用、低可靠、无谱系的组件，保留可复制的有效路径。
4. **有边界感**：承认 agent 的内部推理只是地图，真实执行必须经过外部可审计的实例化层。
5. **可验证**：任何复杂产物都必须能说明“从哪里来、由什么组成、被复用过多少次、是否产生真实因果效果”。

一句话概括：

> Harness 应当通过组装理论的选择机制，让可靠组件被递归复用、层层堆叠，从而演化出高组装指数的复杂能力；同时通过抽象谬误的边界意识，确保这些复杂能力始终被外部制图者、转导通道和因果闭环约束，而不把地图误认为疆域。

## 二、理论背景

### 2.1 组装理论：复杂产物必须有历史

组装理论关注的问题是：为什么某些复杂物体能够真实存在，而不是只停留在可能性空间中。它认为，一个复杂对象的存在并不是偶然的。对象越复杂，就越依赖此前已经形成的部件、路径、选择机制和制造记忆。

其中两个概念对 harness 设计最重要：

- **组装指数（Assembly Index）**：从基础构建单元到目标产物所需的最少递归组合步数。它衡量的是一个产物背后的最短因果历史。
- **拷贝数（Copy Number）**：某个对象或组件被重复制造、调用、实例化或复用的次数。它证明系统中存在某种可靠机制，能够稳定地产生该对象。

组装理论的关键洞察是：一个高组装指数、高拷贝数的对象，几乎不可能凭空随机出现。它必然依赖一条被选择和保留下来的制造谱系。换到 harness 语境中，复杂 agent、复合工具链、多 agent 协作流程和高质量报告都不应被视为一次性生成物，而应被视为已有组件、策略和评估机制递归组合后的产物。

### 2.2 抽象谬误：地图不是疆域

“抽象谬误”关注的是另一个问题：符号计算为什么不能自动等同于真实经验或真实行动。计算系统内部处理的是符号、状态、token、向量和抽象规则；这些东西只有在外部制图者或明确转导机制赋义之后，才成为“关于某物”的地图。

因此，agent 的内部声明并不等于真实世界事实。例如：

- agent 说“我已经发送邮件”，不等于邮件服务器真的有发送记录；
- agent 说“我理解这个系统”，不等于它能在真实环境中正确干预该系统；
- agent 生成一个复杂计划，不等于该计划已经被执行或被验证；
- agent 通过传感器读数构建内部表征，不等于它获得了未经中介的物理现实本身。

对 harness 来说，这意味着必须区分两层：

- **模拟层**：规划、推理、预测、生成、解释、内部世界模型。
- **实例化层**：真实 API 调用、文件修改、硬件动作、外部系统状态变化、可审计日志。

Harness 的责任不是假装 agent 的地图就是疆域，而是清楚记录地图如何生成、由谁赋义、经过什么转导、何时进入真实执行，以及执行结果如何反馈回系统。

### 2.3 两套理论的互补关系

组装理论解决“复杂能力如何可信地出现”；抽象谬误解决“符号能力如何不越界地作用于真实世界”。两者形成互补框架：

| 维度 | 组装理论 | 抽象谬误 |
| --- | --- | --- |
| 核心问题 | 复杂产物如何演化而来 | 符号系统为何不能等同于物理实在 |
| Harness 定位 | 选择环境与物理记忆系统 | 地图系统与实例化边界管理器 |
| 设计重点 | 组件复用、谱系追踪、选择压、拷贝数 | 显式制图者、模拟/实例化隔离、因果闭环 |
| 主要风险 | 无历史的高复杂度随机产物 | 无外部锚定的语义真空 |
| 典型防线 | DAG、调用计数、谱系日志、淘汰机制 | 审计日志、转导记录、真实执行验证、物理主义检验 |

因此，完整的 harness 设计必须同时满足两类要求：复杂能力必须有历史，符号行为必须有锚定。

## 三、Harness 的设计宗旨

基于上述理论，harness 的设计宗旨可以表述为：

> 构建一个具备物理记忆、谱系追踪和实例化边界的选择环境，使高组装指数的复杂 agent、工具和策略只能通过递归复用已有被验证组件的路径涌现，并通过高拷贝数、完整谱系和真实因果闭环证明其可靠性。

这一定义包含三层含义：

1. **复杂能力不能凭空涌现**：任何高阶 agent、工作流或策略，都应能回溯到基础工具、已有组件和历史验证记录。
2. **可靠性来自复用与选择**：一个组件越常被复用、越常在不同任务中通过验证，它越有资格成为更高层能力的构建单元。
3. **执行必须穿过边界**：内部推理只有通过可审计的实例化层产生外部状态变化，才能被视为真正行动。

## 四、概念映射

| 组装理论 / 抽象谬误概念 | Harness 设计映射 |
| --- | --- |
| 基本构建单元 | 原子工具、API、基础策略、原始数据、评估器、执行器 |
| 组装空间 | harness 定义的合法组合规则、编排语法、工具链协议、agent 通信规范 |
| 组装指数 | 从基础单元到复杂产物所需的最短依赖路径长度 |
| 拷贝数 | 组件、策略、流程被调用、实例化、引用、复用的次数 |
| 构造器 | harness 本身、运行其上的 agent、工具执行环境、评估系统 |
| 选择机制 | 组件留存、淘汰、变异、组合、晋升与降级机制 |
| 制图者 | 人类、真实环境、外部验证器、传感器、业务规则或物理约束 |
| 地图 | agent 的内部推理、计划、token 表征、模拟结果、世界模型 |
| 疆域 | 真实 API、文件系统、数据库、硬件、用户确认、外部可观测状态 |
| 转导 | 原始输入到符号、符号到执行动作之间的映射过程 |
| 因果闭环 | agent 决策产生真实状态变化，并由外部反馈重新进入 harness |

## 五、两层架构：组装层与实例化层

### 5.1 组装层：让复杂能力有历史

组装层负责回答：“这个复杂能力是如何被制造出来的？”

它应保存以下信息：

- 组件注册记录：工具、策略、模板、agent、评估器的来源与版本。
- 依赖关系：复杂工作流依赖哪些基础组件，依赖路径有多深。
- 调用与复用次数：哪些组件被大量复用，哪些只是一次性产物。
- 谱系日志：每个复杂产物的父组件、变异来源、验证记录和生成上下文。
- 选择结果：哪些组件被保留、淘汰、晋升为基础设施，哪些进入 graveyard。

组装层的健康状态表现为“金字塔结构”：底部是大量低组装指数、高拷贝数的基础组件；中间是由基础组件递归组合出的策略和流程；顶部是少量高组装指数、可解释、可复现的复杂 agent 或工作流。

### 5.2 实例化层：让符号行为不越界

实例化层负责回答：“这个 agent 说它做了某事，真实世界中是否真的发生了对应变化？”

它应保存以下信息：

- 输入转导记录：原始日志、传感器、用户请求或外部事件如何变成 agent 可处理的符号。
- 输出转导记录：agent 的文本、计划或动作请求如何变成 API 调用、文件修改或硬件动作。
- 执行日志：每次真实执行的时间、参数、目标系统、结果和失败原因。
- 外部反馈：真实环境返回的状态、错误、审计记录或用户确认。
- 边界标记：哪些内容只是模拟，哪些已经进入真实执行层。

实例化层的健康状态表现为“声明可对账”：agent 的每个行动声明都能与外部日志对应；每个内部判断都能说明其输入来源；每个执行结果都能回流为下一轮选择压力。

## 六、核心设计原则

### 6.1 物理记忆原则

Harness 必须保存历史，而不是只保存当前状态。工具调用记录、成功策略、失败案例、评估结果和组件版本都应成为系统记忆的一部分。没有历史记录的复杂产物，即使表现良好，也不能被视为可靠基础设施。

**评估指标**：谱系完整率、历史折叠度、可回溯父组件比例。

### 6.2 递归复用原则

复杂能力应优先由已有组件递归组合，而不是每次从零生成。一个高阶 agent 的可信度，不取决于它看起来多复杂，而取决于它是否复用了足够多经过验证的低阶构件。

**评估指标**：组件复用率、跨任务复用次数、基础设施组件占比。

### 6.3 拷贝数信任原则

高拷贝数是可靠性的 proxy。被大量调用、被不同 agent 复用、被多个项目依赖的组件，更适合成为高阶产物的“原子”。低拷贝数组件可以探索，但不应直接支撑关键路径。

**评估指标**：平均拷贝数、核心组件幂律分布、低拷贝关键依赖数。

### 6.4 选择压原则

Harness 必须主动筛选，而不是无限堆积组件。长期无人复用、无法通过验证、缺乏谱系或成本高于收益的组件，应被降级、归档或淘汰。

**评估指标**：留存率、淘汰速率、变异成功率、递归收益。

### 6.5 反随机涌现原则

高组装指数但低拷贝数、无谱系的产物，应被标记为不可信。它们可能只是一次性拼接出的“玻尔兹曼大脑”式复杂性：外观复杂，但没有可靠制造历史。

**评估指标**：生成不可能性指数、无谱系高复杂产物数、随机涌现拦截数。

### 6.6 显式制图者原则

Harness 必须明确谁或什么机制在赋义。原始输入如何被解释为符号、agent 输出如何被解释为动作，都不能依赖隐式假设。制图者可以是人类、传感器校准规则、业务规则、外部验证器或真实环境反馈。

**评估指标**：制图者显式度、外部锚定接口比例、无锚定感知通道数。

### 6.7 模拟—实例化隔离原则

Harness 必须清楚区分内部模拟与真实执行。规划、草稿、预测、dry-run、世界模型都属于地图；API 调用、文件写入、数据库变更、硬件动作才属于实例化层。

**评估指标**：隔离健全度、行动声明对账率、未验证执行声明数。

### 6.8 因果闭环原则

Agent 的每个重要决策都应通过实例化层产生可观测结果，或明确标记为未执行。反馈必须来自真实环境，而不是内部想象。

**评估指标**：因果闭环率、真实反馈占比、外部状态变化覆盖率。

### 6.9 转导透明原则

输入从物理状态变成符号、输出从符号变成动作的过程必须可审计、可替换、可复现。不能因为系统接入了摄像头、日志或 API，就默认 agent “看见了”或“理解了”真实世界。

**评估指标**：转导透明度、转导可逆性、原始输入保留率。

### 6.10 机制确定性原则

同一物理输入在同一 harness 规则下应被稳定解释。若相同日志、相同事件、相同状态在不同上下文中被任意映射为不同含义，系统就存在语义漂移。

**评估指标**：语义漂移指数、计算身份稳定性、跨环境解释偏差。

### 6.11 反涌现谦逊原则

Harness 不应假定 agent 会通过规模增长自动涌现出理解、意图或真实语义。复杂性应服务于更精确的地图，而不是让地图冒充疆域。

**评估指标**：涌现幻觉拦截率、意图溯源率、无法追溯目标数。

### 6.12 物理主义检验门原则

Agent 的知识、信念或理解不能仅凭内部 token 序列确认。关键声明必须通过真实环境中的行为测试、外部系统日志或用户确认来验证。

**评估指标**：物理主义通过率、声明验证率、关键任务真实执行成功率。

## 七、量化评估框架

### 7.1 组件复用率

```text
拷贝数 C = 组件被调用次数
        + 被其他组件依赖次数
        + 跨 agent / 跨项目复用次数
```

**用途**：衡量组件是否已经成为可靠基础设施。

**健康信号**：核心组件的拷贝数呈幂律分布，少数基础组件被大量复用，多数组件处于探索或淘汰状态。

### 7.2 组装指数

```text
组装指数 AI = 从基础单元到目标产物的最短依赖路径长度
            ≈ DAG 中根节点到目标节点的最长有效路径
```

**用途**：衡量复杂产物的历史深度。

**健康信号**：高 AI 产物应依赖高 C 子组件；若高 AI 产物的所有子组件 C=1，则应标记为未验证复杂性。

### 7.3 谱系完整性

```text
谱系完整率 = 可回溯父组件数 / 理论应有父组件数
历史折叠度 = 构建时复用已有组件数 / 总组件数
```

**用途**：判断 harness 是否真正把历史折叠进当下。

**健康信号**：高 AI 产物的历史折叠度应接近 1。

### 7.4 生成不可能性指数

```text
不可能性指数 = (AI × 独特组件数 U) / log(平均拷贝数 C_avg + 1)
```

**用途**：识别外观复杂但缺乏复用基础的随机产物。

**健康信号**：超过阈值但缺乏谱系日志的产物，应进入人工审查或隔离区。

### 7.5 选择压强度

| 指标 | 计算方法 |
| --- | --- |
| 留存率 | 新组件在 T 周期后仍被使用的比例 |
| 变异成功率 | 对已有组件组合或微调后产生有效新组件的比例 |
| 淘汰速率 | 低拷贝数组件从活跃系统中消失的平均时间 |
| 递归收益 | 复用已有组件相对于从零构建的成本收益比 |

**用途**：判断 harness 是否真的在筛选，而不是只堆积。

### 7.6 跨环境组装一致性

```text
跨环境偏差 = |AI_env1(产物 X) - AI_env2(产物 X)| / AI_env1(产物 X)
```

**用途**：验证 harness 的组装空间是否定义清晰。

**健康信号**：同一产物在不同实例、租户或版本中的 AI 偏差应接近 0。

### 7.7 制图者显式度

```text
制图者显式度 = 有明确外部锚定的接口数 / 总接口数
```

**用途**：判断符号输入输出是否有明确赋义来源。

**红线**：完全来自 agent 自我对话、无外部输入或验证的“感知”，制图者显式度为 0。

### 7.8 实例化隔离健全度

```text
隔离健全度 = 被外部可观测/可审计的真实执行数 / agent 声称的已执行数
```

**用途**：防止 agent 把计划、模拟或草稿误报为真实执行。

**审计方法**：对每次“已执行”声明检查外部日志，如邮件服务器、文件系统、数据库或 API 返回记录。

### 7.9 因果闭环率

```text
因果闭环率 = 产生外部可观测状态变化的决策数 / 总决策数
真实反馈占比 = 来自真实环境的反馈信号数 / 总反馈信号数
```

**用途**：判断 agent 是否只是在内部空转模拟。

### 7.10 转导透明度

```text
转导透明度 = 有完整物理状态到符号映射记录的通道数 / 总 I/O 通道数
转导可逆性 = 能从符号和映射表还原原始状态的比例
```

**用途**：确保输入输出边界可审计。

### 7.11 语义漂移指数

```text
语义漂移指数 = 同一物理输入在不同时间或任务中被映射为不同符号的频率
```

**用途**：发现同一信号在不同上下文中被任意解释的问题。

### 7.12 物理主义通过率

```text
物理主义通过率 = 通过真实行为测试验证的内部状态声明数 / 总声明数
```

**用途**：验证 agent 的“理解”是否能转化为真实干预能力。

## 八、实施路线图

### 阶段一：建立最小可观测底座

优先实现两件事：

1. **DAG 依赖图**：记录工具、策略、agent、工作流之间的依赖关系。
2. **组件使用计数**：记录每个组件被调用、复用、引用和组合的次数。

这两项足以支撑大部分组装理论指标，也是后续选择压和谱系追踪的基础。

### 阶段二：引入谱系与选择机制

在每次产物生成时记录：

- 使用了哪些父组件；
- 父组件版本是什么；
- 是否有验证记录；
- 是否产生了新组件或新策略；
- 新组件是否被后续复用。

同时建立组件生命周期：探索、活跃、基础设施、弃用、graveyard。

### 阶段三：建立模拟—实例化边界

为所有真实动作建立执行网关：

- agent 的内部计划默认标记为模拟；
- 只有通过执行网关的动作才标记为实例化；
- 每个实例化动作必须有外部日志；
- 执行结果必须回写到 harness 的记忆系统。

### 阶段四：建立审计与告警机制

应对以下情况自动告警：

- 高 AI、低 C、无谱系产物；
- agent 声称执行但无外部日志；
- 同一输入在不同上下文中语义漂移严重；
- 新目标无法追溯到人类指令、业务规则或外部约束；
- 关键组件长期低复用却仍处于核心路径。

## 十、结论

用组装理论指导 harness 设计，核心不是追求某个 agent 一次性完美，而是构建一个能够让可靠能力递归生长的选择环境。高阶 agent、工具链和工作流必须携带可验证的组装路径；被大量复用的组件才有资格成为更高层能力的构建单元；看似复杂但缺乏谱系和复用基础的产物，应被视为不可信复杂性。

用抽象谬误约束 harness 设计，核心不是否认 agent 的推理能力，而是拒绝把推理地图误认为物理疆域。Agent 的计划、声明和内部状态必须通过显式制图者、透明转导层和真实因果闭环，才能成为可验证的行动或知识。

因此，一个完整的 harness 应同时是两种系统：

1. **选择环境**：保存历史、递归复用、施加选择压，持续制造高组装指数、高拷贝数能力。
2. **制图工坊**：明确赋义来源、隔离模拟与实例化、审计每次真实执行，防止符号系统自我赋义和越界声明。

这样的 harness 才能支持复杂 agent 能力的长期演化，同时保持工程上的可解释、可审计、可复现和不过度拟人化。
